# Clinician Summary Report v1 — Product Specification

## Intent
A GP-facing summary document delivered via the patient. Designed to be read in ~90 seconds and provide:
- primary pattern + chain
- ranked differential hypotheses (evidence for/against)
- confirmatory tests (suggested, non-prescriptive)
- complete marker appendix for verification

Not a treatment plan. Not a diagnosis.

---

## Output sections

### Header (always present)
- Top-of-document governance statement (2–3 lines, clinical register)
- Footer line (one line on each page)

Required top statement (template; final text governed in contract):
- “Generated by HealthIQ from commercial laboratory results.”
- “Structured discussion aid summarising patterns, hypotheses, and confirmatory testing considerations.”
- “Not a clinical diagnosis or treatment recommendation.”

Footer template:
- “HealthIQ Clinician Summary Report — discussion aid; not diagnostic or prescriptive.”

---

## Page 1 — Structured Summary Block (no paragraphs)

Fixed headings, fixed order:

1) Primary concern (v1: exactly one)
- Derived deterministically from `report_v1.top_findings[0]` only.

2) Key findings (max 5 bullets)
- Bullets derived deterministically from the same primary concern + highest-relevance chain if present.
- No new clinical claims.

3) Chain(s) (max 2)
- Use `report_v1.top_chains` (already ranked).
- Render chain summary text.

4) Top hypothesis (1 line)
- If `root_cause_v1` exists for the primary concern’s signal: show hypothesis #1 title + confidence.
- Otherwise: “No hypothesis set available for this concern in v1.”

5) Confidence + missing data (1 line)
- Derived from `signal_confidence` and missing_data fields (if any).

---

## Page 2 — Hypothesis Ranking

For the primary concern’s root cause block:
- Show ranked hypotheses (max 4 in v1).
- Each hypothesis renders:

Fields:
- Title
- Summary (1–2 sentences)
- Hypothesis confidence (0.0–1.0)
- Ranking rationale line (deterministic template; not frontend-generated)
- Evidence for (bullets)
- Evidence against (bullets)
- Missing data (marker + reason)

Ranking rationale line template (example):
- “Ranked #{rank} because {evidence_for_short}; {evidence_against_or_missing_short}.”

The compiler derives this from:
- top 1 evidence_for item marker_refs if present
- top 1 evidence_against item marker_refs if present, else first missing_data marker_id if present

---

## Page 3 — Confirmatory Tests

Render a short list of confirmatory tests (per hypothesis), but with suppression.

### Confirmatory test suppression rule (hard rule, contract-level)
Suppress a confirmatory test suggestion if ALL are true:
- the test’s `marker_id` exists in the panel input, AND
- the marker has a non-null value, AND
- the marker’s unit is resolved (no unit normalisation error), AND
- the marker has a usable reference range:
  - two-sided OR one-sided range is acceptable if it resolved without error

If a marker failed unit normalisation or range parsing, it does NOT count as present for suppression.

The output contract must include:
- `suppressed_confirmatory_tests: [test_id...]`

Confirmatory tests are framed as suggestions:
- “Consider including…”
- “Consider discussing…”
Never “order” or “you need”.

---

## Appendix — Full Marker Table (always present)
For each biomarker:
- name (canonical id)
- value
- unit
- reference range (including one-sided and banded profile where available)
- lab range source and any effective-from note if present
- in/out of range status (if determinable)

---

## Data Quality Box (always present)
A small box displayed on page 1.

Fields:
- Panel completeness (fraction: present / expected-for-fired-signals)
- Lab range quality (per primary metric of fired top findings):
  - complete / one-sided / banded-profile / missing
- Confidence caveat (one line derived from the lowest signal confidence among `report_v1.top_findings`)

The contract must include:
- `data_quality_passed: bool` (true if all top findings have usable ranges and units and completeness above a threshold)
CSR still renders when false; box becomes more prominent.

---

## Selection rules (contract-level, not frontend)
Spotlight selection in v1:
- Primary concern: `report_v1.top_findings[0]` only.
- Additionally include:
  - top chain by `chain_confidence` from `report_v1.top_chains`
  - any mandatory clinician referral triggers (if present in governed outputs) regardless of rank

Frontend renders what contract provides; does not select.
